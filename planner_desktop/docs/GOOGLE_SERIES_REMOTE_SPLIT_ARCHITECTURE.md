# Phase 3.2B3C1 — Remote "this and future" split of a Planner-owned linked series

Эта фаза добавляет безопасное удалённое разделение связанной серии:
пользователь редактирует чистую Planner-owned связанную серию «с этого
экземпляра и дальше». После завершения в Google существуют **два**
recurring-мастера: исходный, укороченный до экземпляров ДО целевого слота,
и один новый мастер-преемник, начинающийся с целевого слота.

Adoption чужих (внешних) recurring-мастеров сюда **не входит** — это
Phase 3.2B3C2. Автоматической/фоновой синхронизации нет: единственный
сетевой триггер — ручной sync.

## 1. Слои

| Слой | Модуль | Ответственность |
|---|---|---|
| Domain (pure) | `domain/google_series_split.py` | валидация, подсчёт слотов, правила COUNT/UNTIL/never, канонические снапшоты и хэши; ни Qt, ни SQLite, ни Google |
| Storage | `storage/calendar_series_remote_split_store.py` | schema v11 таблица `calendar_series_remote_splits`, durable state machine, локальная атомарная финализация |
| Use case | `usecases/remote_series_split_service.py` | preflight, создание/отмена/rollback плана, блокировки конкурирующих операций |
| Gateway | `sync/google_calendar_gateway.py` | full-resource `update_recurring_master_full` + `insert_split_successor_master` |
| Engine | `sync/calendar_series_remote_split_engine.py` | пошаговое выполнение плана в ручном sync, реконсиляция, rollback |
| Pull | `sync/calendar_sync_engine.py` | split-aware классификация мастеров до обычной B3A conflict-логики |
| UI | `qml/components/RemoteSeriesSplit*.qml` | диалог, статусы, восстановление, история; ноль сетевых вызовов |

## 2. Право на разделение (eligibility)

`validate_split` отклоняет план, если нарушено хотя бы одно условие:

- серия существует, активна и не удалена;
- активная связь есть, `link_status == synced`, generation связи актуальна;
- нет pending и нет terminal операции мастера (`pending_calendar_series_ops`);
- нет неразрешённого конфликта мастера и состояния `remote_deleted`;
- правило серии lossless-представимо в Google RRULE (round-trip);
- целевой `occurrence_key` — реальный слот ИСХОДНОГО расписания
  (проверяется детерминированным генератором, не арифметикой);
- целевой слот НЕ первый экземпляр серии (первый — обычная правка всей
  серии, сервис возвращает структурированный отказ `target_is_first`);
- целевой слот не в прошлом и не после последнего экземпляра;
- вид серии не меняется: timed → timed, all-day → all-day
  (конверсия timed↔all-day отложена);
- у серии со временем таймзона преемника — валидный IANA;
- ни одного будущего (>= target) исключения:
  - локальные exception-строки Task;
  - локальные тумбстоуны;
  - pending/terminal occurrence-операции;
  - synced remote exceptions / `synced_exception`/`pending_*` occurrence-links;
  - remote cancellations;
  - неразрешённый карантин;
  - EXDATE/RDATE в правиле мастера (Planner их не создаёт; наличие —
    признак чужой правки и блокирует split);
- количество экземпляров до целевого представимо как COUNT (<= 999).

Будущие исключения **никогда не удаляются, не переносятся и не
сбрасываются молча**: сервис возвращает точные количества и даты, а
пользователь сперва разрешает их существующими средствами B3A/B3B.
Прошлые (< target) исключения и выполненная история остаются у исходной
серии и split не блокируют.

## 3. Идентичность цели

Целевой слот задаётся неизменяемым `occurrence_key` исходного расписания.
`local_occurrence_to_google_original_start` даёт точный Google
`originalStartTime` (kind `date`/`datetime` + IANA-таймзона), который
сохраняется в плане (`target_original_start_*`) и используется как
provenance-маркер преемника.

## 4. Правила разрезания повторения

Подсчёт — только детерминированным генератором (`generate_occurrences`),
посчитанные слоты, а не календарная арифметика «минус одна секунда».

Исходный мастер (trim):

- DTSTART, title, notes, таймзона и прочие Planner-owned поля сохраняются;
- частота, интервал и BY*-компоненты сохраняются;
- окончание заменяется на `COUNT = occurrences_before_target` (всегда >= 1).

Преемник:

- DTSTART = исходный целевой слот, изменённый запрошенным расписанием
  (дата целевого слота + новое время/длительность или all-day дата);
- правило сохраняет поддерживаемую частоту/интервал/BY* (или заменяется
  запрошенным правилом);
- окончание:
  - исходное never → преемник never;
  - исходное COUNT=N → преемник COUNT = N − occurrences_before_target
    (before < N обязательно);
  - исходное UNTIL → преемник сохраняет исходный lossless UNTIL.

## 5. Идентификаторы преемника

При создании плана резервируется **один** новый series UID
(`reserved_successor_series_uid`, uuid4, фиксируется в плане) и из него
детерминированно выводится remote id
(`deterministic_remote_event_id(reserved_uid)`). Повторные/дублирующие
запросы возвращают тот же план — второй UID/ID не выдаётся. Random
fallback запрещён: коллизия чужого события с этим id — terminal.

## 6. Durable state machine (schema v11)

Таблица `calendar_series_remote_splits` хранит полный план: снапшоты
исходного, обрезанного и желаемого преемника (canonical payload JSON +
SHA-256 хэши), acknowledged ETag исходного мастера, состояние и ETag-и
после каждого удалённого шага. Отдельная audit-таблица не нужна: строка
плана сама фиксирует все переходы (`state`, `*_etag`, `attempts`,
`last_error`, timestamps), а завершённые/откаченные планы остаются
запрашиваемой историей.

Состояния:

```
pending ──(trim ok)──> source_trimmed ──(insert ok)──> successor_created
   │                        │                              │
   │ conflict               │ conflict                     │ (local atomic
   ▼                        ▼                              ▼  finalize)
conflict                 conflict                      completed
   │
   └─(user)→ rollback_pending → successor_removed_for_rollback → rolled_back
pending ──(cancel, 0 Google-вызовов)──> rolled_back
любой шаг ──(dead-letter)──> terminal_error
```

- `local_finalize_pending` — зарезервированное подсостояние successor_created
  (оба мастера верифицированы, идёт локальная транзакция);
- переходы всегда персистятся ДО следующего удалённого шага;
- план не удаляется и не переписывается до completed/rolled_back.

## 7. Выполнение в ручном sync

Порядок цикла: **1) активные split-планы**, 2) очередь мастеров,
3) очередь occurrence, 4) обычные Task-операции, 5) pull, 6) сводка.

`pending`: fetch исходного мастера → проверка Planner-ownership (маркер
`planner_series_uid`), ETag == `source_remote_etag_base` И фактический
канонический хэш == `source_original_payload_hash` → merge обрезанного
recurrence + обновлённых приватных маркеров в ПОЛНЫЙ ресурс (чужие поля —
attendees/reminders/location/conference — сохраняются как есть) → один
`events.update` с If-Match → state `source_trimmed` + возвращённый ETag.
Если мастер уже в точности обрезан (crash-recovery) — реконсиляция без
записи. Любое расхождение — `conflict` без записи.

`source_trimmed`: реконсиляция фактического содержимого исходника →
fetch детерминированного id преемника; если существует и ownership +
фактический канонический хэш совпадают — успех без insert; чужой мастер —
terminal; свои маркеры без совпадения содержимого — НЕ успех (conflict).
Иначе один `events.insert` → state `successor_created` + ETag.

`successor_created`: verify оба мастера (точное содержимое) → одна
локальная SQLite-транзакция финализации → `completed` внутри неё.
Локальный сбой оставляет `successor_created`; следующий sync повторяет
ТОЛЬКО локальную транзакцию, удалённых записей больше нет.

## 8. Локальная атомарная финализация

`finalize_linked_remote_split_atomic(...)` — одна транзакция:

1. ревизия исходной серии равна плановой (разрешены только явные
   local-only мутации: выполнение, теги);
2. исходная TaskSeries обрезается (COUNT before);
3. создаётся TaskSeries-преемник с зарезервированным UID;
4. будущие replaceable (живые, невыполненные, не-exception) занятые
   слоты >= target переназначаются преемнику / пересоздаются
   материализацией;
5. выполненная история и прошлые exception/тумбстоуны не трогаются;
6. теги серии копируются преемнику; occurrence-теги остаются на строках;
7. создаётся synced-связь преемника с его remote-мастером (ETag/hash);
8. связь исходника обновляется на обрезанный ETag/hash/revision;
9. план помечается `completed`.

Обычные Task-операции Calendar НЕ порождаются; материализованные
экземпляры НИКОГДА не выгружаются как ordinary events.

## 9. Race-защита (ETag + фактическое содержимое)

Приватные маркеры сами по себе доказательством не считаются: чужая
правка их не обновляет. Каждая проверка — маркеры **плюс** канонический
хэш фактического содержимого; любое расхождение с acknowledged базой:

- до trim — план `conflict`, снапшот обновлён, записи нет;
- после trim до insert — insert не выполняется, `conflict`;
- после insert до финализации — финализации нет, `conflict` c обоими
  снапшотами.

Выход из `conflict` — только явное действие пользователя: rollback или
отмена/пересоздание плана после разрешения конфликта мастера.

## 10. Rollback

- `pending` (0 удалённых шагов): локальная отмена, ноль Google-вызовов;
- `source_trimmed`: verify исходник = точное Planner-owned обрезанное
  состояние → восстановление полного исходного снапшота текущим ETag →
  `rolled_back`;
- `successor_created`: verify преемник Planner-owned и точный → delete →
  `successor_removed_for_rollback` (persist) → verify исходник обрезан →
  восстановление исходного снапшота → `rolled_back`;
- если любой мастер изменился неожиданно — НИЧЕГО не перезаписывается и
  не удаляется, план `conflict`, требуется ручной разбор;
- remote-успех/локальный сбой: retry реконсилирует каждый завершённый шаг
  по фактическому содержимому — двойных insert/update/delete нет.

## 11. Pull во время split

Split-обработчик выполняется ДО обычной B3A-классификации мастеров:

- исходный мастер в ожидаемом обрезанном состоянии — эхо, обновляется
  ETag плана, конфликт не создаётся;
- мастер с id преемника ассоциируется с зарезервированным UID; он не
  становится ни ordinary Task, ни unowned external master;
- неожиданное содержимое любого из двух — план `conflict`;
- отмена (cancelled) исходника или преемника фиксируется в плане без
  автоматического пересоздания;
- сбой персиста обработчика прерывает pull ДО продвижения курсора.

После completed обе связи живут независимо по обычным правилам
B2/B3A/B3B.

## 12. Блокировки на время активного плана

Разрешено: локальное выполнение occurrence, локальные теги, прочие явно
local-only метаданные. Заблокировано: правка всей серии
(расписание/правило/название/заметки), второй split, connect/disconnect/
delete/recreate связи, schedule-операции occurrence >= target,
разрешение конфликтов того же мастера (пока план сам не станет conflict).

## 13. Границы Phase 3.2B3C1

- attendees, reminders, location, conferencing не управляются
  (сохраняются на исходнике merge-ом; преемник их не получает);
- timed ↔ all-day конверсия отложена;
- миграция будущих исключений на преемник — возможное будущее
  расширение, НЕ входит в B3C1 (они блокируют split);
- adoption внешнего Google-мастера — Phase 3.2B3C2: pull может только
  каталогизировать чужой мастер; связывание/усыновление и split чужих
  серий остаются отключёнными.
